# Rule Evolution Guide

> 하네스의 규칙은 **살아있는 시스템**입니다.
> 실수가 발생할 때마다 규칙을 추가하여 같은 실수가 구조적으로 반복 불가능하게 만듭니다.

---

## Evolution Cycle

```
실수 발생 → 원인 분석 → 규칙 추가 → 게이트 반영 → 검증 → 시스템 강화
    ↑                                                          |
    └──────────────────────────────────────────────────────────┘
```

---

## Step 1: 실수 발생 시

AI 에이전트 또는 개발자가 규칙을 위반한 코드를 작성했을 때:

1. **문제를 기록한다**
   - 어떤 파일에서 어떤 위반이 발생했는가?
   - 왜 발생했는가? (규칙이 없었는가, 규칙이 불명확했는가, 규칙을 몰랐는가)
   
2. **분류한다**
   - **규칙 부재**: 해당 패턴을 금지하는 규칙이 없었음 → 새 규칙 추가
   - **규칙 불명확**: 규칙은 있지만 해석의 여지가 있었음 → 규칙 구체화
   - **게이트 미비**: 규칙은 있지만 자동 검사가 없었음 → 게이트 추가
   - **허위 양성**: 게이트가 정상 코드를 위반으로 판정 → 규칙/게이트 수정

---

## Step 2: 규칙 추가

### 의존성 경계 규칙 추가 (.harness/gates/rules/boundaries.yaml)

```yaml
# 새 규칙 예시
nextjs:
  - from: "src/hooks"
    cannot_import: ["prisma", "pg"]
    reason: "커스텀 훅에서 DB 직접 접근 금지. 서버 액션 경유"
    stacks: [nextjs]
    file_patterns: ["*.ts", "*.tsx"]
```

### 코딩 컨벤션 추가 (docs/code-convention.yaml)

```yaml
# 새 규칙 예시
typescript:
  - id: TS-006
    rule: try-catch에서 error 타입을 unknown으로 받고 타입 가드로 처리한다
    stacks: [typescript]
```

### ADR 추가 (docs/adr.yaml)

구조적 결정이 필요한 경우 ADR로 기록합니다.

---

## Step 3: 게이트 반영

규칙을 추가한 후 게이트가 해당 규칙을 자동으로 검사하는지 확인합니다.

- **boundaries.yaml에 추가한 규칙**: `check-boundaries.sh`가 자동으로 읽어서 검사
- **새로운 유형의 검사가 필요한 경우**: `check-*.sh` 스크립트를 새로 작성

---

## Step 4: 검증

```bash
# 위반 전체 스캔
.harness/detect-violations.sh

# 의도적으로 위반하는 테스트 코드를 작성하여 게이트가 잡는지 확인
echo "import prisma from '@prisma/client'" > /tmp/test-violation.tsx
# → check-boundaries.sh가 이를 감지해야 함
```

## Principles

1. **프롬프트로 부탁하지 말고, 게이트로 강제하라**
   - "DB를 직접 호출하지 마" → 프론트엔드→DB import 시 빌드 실패

2. **규칙은 구체적이고 검증 가능해야 한다**
   - 나쁜 예: "코드를 깔끔하게 작성한다"
   - 좋은 예: "함수는 50줄을 넘지 않는다"

3. **과도한 규칙보다 핵심 규칙에 집중하라**
   - 규칙이 너무 많으면 무시됨. 정말 중요한 것만 게이트화

4. **정기적으로 규칙을 리뷰하라**
   - 분기 1회: 불필요해진 규칙 삭제, 규칙 충돌 해소
   - 규칙도 코드처럼 리팩토링이 필요함

5. **허위 양성은 즉시 수정하라**
   - 게이트가 정상 코드를 막으면 개발자가 게이트를 무시하기 시작함
   - `--no-verify`를 습관적으로 쓰게 되면 하네스의 의미가 사라짐
