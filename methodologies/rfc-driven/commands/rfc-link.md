# /rfc-link — Link Working Code to an RFC

> Tell the gate "this big change is OK because RFC X authorized it."

## When to use

- Working on a PR that exceeds the RFC threshold (>500 LOC per file or >1000 total)
- File touches a path in `always_required_paths` (auth, migrations, etc.)
- Want to suppress the rfc-driven gate warning for a legitimate change

## Usage

```
/rfc-link <rfc-id> --files F1 [F2 ...] [--modules M1 [...]]
/rfc declare-pr --rfc-id RID --pr-files F1 [F2 ...]
```

Both forms update the same data — `link` is more general (modules + files), `declare-pr` is specific to "the files in my current PR".

## Prerequisites

- Target RFC must be in `accepted` state (script blocks otherwise)
- File paths should be repo-root relative

## Instructions

### Step 1 — Find the relevant accepted RFC

```shell
python3 .harness/methodologies/rfc-driven/scripts/rfc.py list --status accepted
```

If no relevant RFC exists, **stop** and create one — don't link to a tangentially related RFC.

### Step 2 — Link files

```shell
# Add specific files
python3 .harness/methodologies/rfc-driven/scripts/rfc.py \
  link rfc-2026-04-30-eventbus-replacement \
  --files src/events/publisher.ts src/events/consumer.ts

# Add a whole module
python3 .harness/methodologies/rfc-driven/scripts/rfc.py \
  link rfc-2026-04-30-eventbus-replacement \
  --modules src/events/

# Both at once
python3 .harness/methodologies/rfc-driven/scripts/rfc.py \
  link rfc-2026-04-30-eventbus-replacement \
  --files src/main.ts \
  --modules src/events/ src/messaging/
```

### Step 3 — Declare current PR's files (convenience)

If you're working on a feature branch and want to link all currently changed files:

```shell
# List changed files in your branch
git diff --name-only main...HEAD > /tmp/pr-files.txt

# Declare them
python3 .harness/methodologies/rfc-driven/scripts/rfc.py \
  declare-pr \
  --rfc-id rfc-2026-04-30-eventbus-replacement \
  --pr-files $(cat /tmp/pr-files.txt)
```

### Step 4 — Verify the gate now allows the change

```shell
.harness/methodologies/rfc-driven/gates/check-rfc-required.sh
```

Linked files should no longer warn. Files NOT covered by the RFC will still warn.

## How the gate uses links

The gate reads `.harness/rfc-driven/.rfc-links.yaml` (auto-generated by `link`):

```yaml
schema_version: 1
files:
  src/events/publisher.ts: ["rfc-2026-04-30-eventbus-replacement"]
  src/events/consumer.ts:  ["rfc-2026-04-30-eventbus-replacement"]
modules:
  src/events/: ["rfc-2026-04-30-eventbus-replacement"]
```

Files matched by the RFC's governs_files OR governs_modules are exempt from the LOC threshold.

## Composition

| Combo | Effect |
|-------|--------|
| `+ parallel-change` | RFC link covers entire migration plan, not individual phases |
| `+ strangler-fig` | RFC governs the new module + facade |
| Multiple RFCs | A file can be governed by >1 RFC (typically successive versions) |

## Constraints

- Only `accepted` RFCs accept links (script blocks others)
- Cannot link to `superseded` RFCs (use the superseding RFC)
- Glob patterns aren't supported in `--files` — list explicit paths or use `--modules` for directories
- Modules are matched as path prefixes (e.g., `src/events/` matches `src/events/x/y.ts`)

## Anti-patterns

- ❌ **Linking everything to one mega-RFC** to suppress gates — defeats the purpose
- ❌ **Linking files unrelated to the RFC's design** — silently corrupts traceability
- ❌ **Linking after the merge** — link before commit so gate validates
- ❌ **Generic RFCs to "cover anything"** — RFCs should be specific decisions

## Failure modes

| Symptom | Cause | Fix |
|---------|------|-----|
| `can only link files to 'accepted' RFCs` | RFC not accepted | Run `/rfc accept` first |
| Gate still warns on linked file | Path mismatch (relative vs absolute) | Use repo-root relative paths consistently |
| `.rfc-links.yaml` doesn't update | Manual link skipped script | Always use `/rfc link` (script regenerates the file) |
| Wrong RFC linked | Sloppy linking | `link` doesn't unlink — manually edit yaml or re-link with correct RFC |
